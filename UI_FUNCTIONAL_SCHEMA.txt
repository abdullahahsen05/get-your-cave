UI_FUNCTIONAL_SCHEMA

Project: GETYOURCAVE
Purpose: Document the current hardcoded frontend structure so the app can later be wired to PostgreSQL, Prisma, Socket.IO, Stripe, file uploads, contracts, and authentication without changing the existing design.

Important constraint for future implementation:
- Preserve all current spacing, colors, layout, typography, borders, shadows, cards, button shapes, responsive behavior, and overall visual composition.
- Replace only hardcoded data with dynamic data.
- Do not use another page’s temporary state as the source of truth. Every page must fetch its own data using the URL, session, or API.

============================================================
1. PROJECT OVERVIEW
============================================================

Current frontend impression:
- The app is a premium storage marketplace with a strong architectural / minimalist brand style.
- The public site supports discovery of storage spaces, trust messaging, and conversion to owner listing.
- The application already visually includes the major business areas needed for the marketplace:
  - Landing page and marketing sections
  - Browse/search listings page
  - Listing detail page
  - Owner dashboard
  - Renter dashboard
  - Admin dashboard
  - Signup / onboarding flow
  - Verification document upload page
  - Listing creation flow
  - Messaging / chat UI
  - Contracts UI
  - Invoices UI

Business flow currently implied by the UI:
1. Public user lands on the homepage.
2. Public user browses listings and opens a listing detail page.
3. Renter books a listing and later pays invoices.
4. Owner creates a listing, manages bookings, views payouts, and signs contracts.
5. Both parties exchange messages in chat.
6. Users upload verification documents.
7. Contracts are generated, previewed, signed, and downloaded.
8. Admin monitors platform activity, listings, users, and revenue.

What is still hardcoded:
- Counts, totals, dates, names, addresses, locations, and status labels are all sample values.
- Images are fixed assets / remote placeholders.
- Tables, cards, and timelines are currently static.
- No page is yet wired to database queries or API endpoints.
- There is no visible login page route in the current file set.

14. PAGE INDEPENDENCE PLAN
============================================================

Principle:
- Every page should fetch its own data using the route, session, and its own API call.
- Do not pass critical data from one page to another through temporary UI state.

Recommended independence by page:
- /:
  - fetch featured listings and stats directly from public APIs
- /storage:
  - derive filters from query params and fetch matching listings
- /storage/listing-details:
  - load by listing ID from URL and fetch listing data independently
- /signup:
  - local form state only, then auth API
- /create-listing:
  - load draft from backend if editing, otherwise start empty
- /owner/dashboard:
  - fetch owner data from session user id
- /renter/dashboard:
  - fetch renter data from session user id
- /admin/dashboard:
  - fetch admin aggregates from admin API
- /messaging:
  - fetch conversation list from session user id and selected conversation id from route or local selection
- /contracts:
  - fetch contracts from session user id and role
- /invoices:
  - fetch invoices from session user id and filters in query params
- /document:
  - fetch current user documents from session user id

Bad pattern to avoid:
- Listing detail depending on a listing object passed from a card click.
- Dashboard depending on prior page state.
- Chat depending on hidden local state from a previous screen.

Good pattern:
- The route identifies the record, and the page fetches the data itself.
